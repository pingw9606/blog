# 07 · 后端查库找问题的流程（Java / MySQL 实战）

> 重点：**运维直连数据库 ≠ 后端正常查库**。下面是后端工程师排查线上问题的真实流程。

## 一、两种"查库"要分清

| | 应用查询（正常业务） | 直连数据库（调试/修补） |
|---|---|---|
| 谁用 | 每个用户请求 | 工程师临时操作 |
| 方式 | ORM + API（JPA/MyBatis） | GUI 客户端 / `mysql` 命令行 |
| 有无权限/校验 | ✅ 有 | ❌ 裸操作 |
| 防注入/类型安全 | ✅ | 看你怎么写 |
| 适合 | 跑业务 | 核对数据、补数据、紧急修复 |

## 二、排查线上问题的标准流程（从外到内）

**1. 从现象 + 日志入手（最关键）**
- 用户报错 / 监控告警 → 去**应用日志**（Kibana/ELK 或服务器上 logback 日志）。
- 按时间、`traceId`、用户 ID 搜索，定位：哪个接口、什么异常、入参是什么。

**2. 看接口实际跑了什么 SQL**
- JPA/MyBatis 打开 **SQL 日志**：
  ```properties
  # application.yml / properties
  logging.level.org.hibernate.SQL=DEBUG          # JPA
  mybatis.configuration.log-impl=...StdOutImpl    # MyBatis
  ```
- 线上不重启看：用 **Arthas**（阿里诊断神器）attach 进程，看方法入参/返回/调用栈。

**3. 直连数据库核对数据**
- 工具：**DBeaver / Navicat / IDEA Database**，或命令行 `mysql`。
- 跑 `SELECT ... WHERE ...` 核对数据本身对不对。
- 这步就是本项目里我帮你用 `psql` 做的事（改角色、查用户）。

**4. 慢 / 卡：执行计划 + 索引**
```sql
-- 找慢 SQL：开慢查询日志
SET GLOBAL slow_query_log = ON;
-- 分析单条 SQL 走没走索引
EXPLAIN SELECT * FROM user WHERE email = 'x@y.com';
```
看是否全表扫描、有没有用索引 → 加索引 / 改写 SQL。

**5. 监控与链路**
- Grafana/Prometheus：QPS、慢查询数、连接数、CPU/内存。
- SkyWalking/Pinpoint：一个请求跨服务的完整耗时链路。

## 三、线上铁律
- 查库用**只读账号**、尽量连**从库**，别在主库跑重查询。
- `SELECT` 一律带 `LIMIT`，避免拉爆内存。
- **改数据走工单/审批**，禁止随手 `UPDATE/DELETE`（容易误伤、无法回滚）。
- 改前先 `SELECT` 确认影响行数，`UPDATE` 一定带精确 `WHERE`。

## 四、在练手项目里模拟同一套流程
原理一致，可迁移到公司 MySQL：
1. **开 ORM 的 SQL 日志**，在网站点操作，观察"ORM 方法 → 真实 SQL"。
2. **用 DBeaver 连库**（同时支持 PG 和 MySQL）点查练手。
3. 对查询跑 **`EXPLAIN`** 看执行计划。
4. 制造一个报错，去服务端日志按错误栈定位到具体接口和行号。

## 五、常用排查命令速查（服务器侧）
```bash
# 应用进程/端口
ps -ef | grep java        # 找 Java 进程
ss -tlnp | grep 8080      # 看端口监听
# 资源
df -h                     # 磁盘（满了会引发各种诡异问题）
free -h                   # 内存
top / htop                # CPU/内存占用
# 日志
tail -f /path/app.log     # 实时看日志
grep 'ERROR\|Exception' app.log | tail -50
```
