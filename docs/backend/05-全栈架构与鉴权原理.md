# 05 · 全栈项目结构与登录鉴权原理

以「欧阳一族」小说站为例（Nuxt + Prisma + PostgreSQL）。

## 一、项目分层（和 Java 后端对照）

| 本项目 | Java(Spring) 对照 | 职责 |
|--------|------------------|------|
| `pages/*.vue` | 前端模板 | 页面/路由 |
| `server/api/*.ts`（handler） | Controller | 接口入口、参数校验 |
| `server/utils/guards.ts` | 拦截器/AOP 鉴权 | 权限校验 |
| `server/utils/*.ts` | Service | 业务逻辑 |
| `prisma`（ORM） | JPA/MyBatis | 数据访问 |
| `prisma/schema.prisma` | Entity + Flyway | 数据模型/迁移 |

## 二、登录鉴权用的什么方案
**`nuxt-auth-utils` + 加密 Cookie 会话**。注意：**不是 JWT，也不是服务端 session 表**——用户身份被**加密后塞进 httpOnly Cookie**，密钥是环境变量 `NUXT_SESSION_PASSWORD`。服务端不存会话状态（无状态）。

## 三、完整链路
```
浏览器 填账号密码
  │ POST /api/auth/login
  ▼
服务端：按邮箱/用户名查库 → verifyPassword(哈希,明文)
  │ 校验通过 & 未封禁
  │ setUserSession() 把 user 加密进 cookie
  ◄── Set-Cookie: 加密会话
之后每次请求浏览器自动带上 cookie → getUserSession() 解密认人
```

## 四、关键代码点
- **密码存哈希**：注册时 `hashPassword(password)`（scrypt），数据库只存 `passwordHash`，绝不存明文。
- **登录校验**：`verifyPassword(user.passwordHash, 明文)`。
- **权限守卫**（`guards.ts`）：`requireAuth` / `requireAuthor` / `requireAdmin`，**每次都回数据库查最新 role/status**，不信任 cookie 里的旧角色。

## 五、两个值得学的设计
1. **角色实时生效**：因为守卫每次查库，所以"读者→审批为作者"后**不用重登**，下次请求角色就升级了。
2. **安全细节**：
   - 登录失败统一报"账号或密码错误"，**不暴露账号是否存在**（防枚举）。
   - 封禁用户即时失效（查库发现 banned 就清会话）。
   - 会话 cookie 是 httpOnly + 加密，前端 JS 偷不到、改不了角色。

## 六、一句话总结
**密码存哈希；登录后把身份加密进 httpOnly Cookie，浏览器每次自动带上；服务端解密认人，并每次查库核对最新角色和封禁状态。** 既无状态又能让权限实时生效。

> JWT vs Cookie-Session 对照：JWT 把信息签名后放 token（前端存），本方案把信息加密放 cookie（浏览器自动带）。Java 里 Spring Security 两种都常见。
