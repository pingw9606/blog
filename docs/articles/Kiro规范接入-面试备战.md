# Kiro AI 研发提效规范接入 —— 面试备战文档

> 项目：vibeshort-website（Next.js 16 + React 19 + TS 海外官网）
> 你的角色：前端负责人，主导把团队的 AI 研发提效规范（kiro-standards）接入并完成前端适配
> 用途：面试时讲清楚"做了什么、为什么这么做、技术难点、价值产出"

---

## 一、一句话电梯陈述（先背熟）

> "我们后端负责人设计了一套基于 Kiro IDE（steering/hooks/skills 机制）的 AI 研发提效规范，先在后端项目跑通。**我负责把它改造适配到我负责的 Next.js 官网项目**——原版规则全是面向 Java 后端的，我逐条理解其质量意图、重写成前端等价规则，并补充了 SEO、TS 类型等前端特有维度，把编码规范、自动 Code Review、提测自动化、经验沉淀、发布计划、效能度量这条全链路在前端项目里跑通。目标是开发者正常写代码、正常提交，质量关卡和重复工作由 AI 后台自动完成，**零额外操作成本**。"

---

## 二、这套规范到底是什么（背景知识）

它是一套挂在 **Kiro IDE** 上的"AI 研发流程自动化"约定，核心是三类文件 + 两类自动化触发器：

| 机制 | 目录 | 作用 | 加载方式 |
|------|------|------|----------|
| **steering** | `.kiro/steering/*.md` | AI 规则，约束 AI 的行为（编码规范、CR 规则、提交规范等） | 每次对话自动常驻加载 |
| **skills** | `.kiro/skills/*.md` | 大文档能力包（评审清单、发版 SOP 等） | 按需触发才加载，不常驻占用上下文 |
| **hooks** | `.kiro/hooks/*.json` | 事件驱动自动化（git commit 前后强制触发） | 在 Kiro IDE 的 Agent Hooks 面板注册 |
| **learnings** | `.kiro/learnings/` | 经验自动沉淀，随 git 同步全团队 | AI 写代码前自动检索复用 |
| **specs** | `.kiro/specs/` | 结构化需求（requirements/design/tasks） | Spec 驱动开发 |

**全链路：** 需求评审(11维度) → 技术评审(17维度) → 编码(规范自动执行) → 提交(自动 CR+经验沉淀) → 提测(自动合并+飞书通知) → 发布(15维度校验)。

**关键设计思想：**
- **steering vs skills 分离**：常驻规则要短（省 token、每轮都加载），重文档按需加载，避免上下文爆炸。
- **hooks 是"强制触发"**：不依赖 AI 自觉，git commit 这个动作必然触发 pre-commit CR 和 post-commit 沉淀。
- **规范即代码**：规范本身放在 GitLab 仓库（`mid-platform/kiro-standards`），通过"初始化/更新 kiro 规范"指令同步，版本化管理。
- **个性化与通用分离**：通用规范全团队统一，项目特有的检查项放 `*-custom.md`，同步时不被覆盖。

## 三、我具体做了什么（你的工作产出）

### 1. 规范接入
- 从 `mid-platform/kiro-standards` 全量拉取规范到项目 `.kiro/` 目录（clone + 复制），保留项目原有的 specs、settings、已装的 skill。

### 2. 前端本地化（核心工作量，原规范是 Java 后端的）

| 文件 | 后端原版 | 我改成的前端版 |
|------|---------|---------------|
| `coding-standards.md` | 分层(Controller/Service/Mapper)、try-catch、SQL 注入、N+1 查询、事务 | 组件分层、DOMPurify 消毒、server-only 边界、并行取数防 N+1、next/image、TS 严格、i18n/路由红线 |
| `code-review.md` | MyBatis `${}`、@Transactional、分布式锁 | 6 类检查项：规范/安全/性能/业务/**SEO(canonical/hreflang)**/类型构建 |
| `commit-convention.md` | 触发文件 `*.java,*.xml,*.go` | `*.ts,*.tsx,*.js,*.jsx,*.mjs,*.css,*.scss` |
| `merge-to-test.md` | `mvn clean compile` / `mvn test` | `npm run lint && npm run build`（无单测则降级为构建门禁） |
| `reactive-rules.md` | 敏感文件 pom.xml/Mapper.xml/Jenkinsfile | next.config/.env/package.json/i18n/sitemap/lib-api |
| `meegle-config.md` | — | 保留飞书项目 project-key，应用名/提测分支按本仓库配置 |

### 3. 环境与配置
- 安装并授权 **meegle CLI**（飞书项目 Meego 的命令行，让 AI 能校验需求/Bug 编号、查测试人员、@人）。
- 创建远程 `test` 提测分支；对齐本地 `git user.name` 与 GitLab username（提测 @人/效能统计靠它识别开发者）。
- 更新 `.gitignore` 忽略规范运行产物。

### 4. 提交
- 规范化提交 `[docs] 接入 kiro-standards 前端规范并本地化` 并推送远程。

## 四、高频面试问题 & 参考答案

### Q1：这套东西解决了什么问题？为什么要做？
A：传统研发流程里，Code Review、提测通知、经验沉淀、发布计划这些都靠人自觉，质量参差、容易遗漏、沟通成本高。我们把这些固化成 AI 自动执行的规则：开发者只管写代码和提交，AI 在后台自动做质量关卡和重复工作。**核心价值是把"靠人自觉"变成"流程强制 + 零额外操作成本"**。

### Q2：steering、skills、hooks 有什么区别？为什么这样分？
A：
- **steering** 是常驻规则，每轮对话都加载，所以必须短小精悍，否则浪费上下文 token。放编码规范、提交规范这类高频规则。
- **skills** 是按需加载的大文档（比如 17 维度技术评审清单），只在触发词出现时才加载，避免常驻占用上下文。
- **hooks** 是事件驱动的强制触发器，绑定在 git commit 前后，不依赖 AI"记得"去做。
这套分层本质是**上下文成本与强制力的权衡**：高频短规则常驻、低频重文档按需、关键动作强制。

### Q3：前端本地化最难的点在哪？
A：难点不在"翻译规则"，而在**理解每条后端规则背后的质量意图，再映射到前端等价物**。举例：
- 后端的"N+1 查询"→ 前端是"组件内逐项发请求"，对应方案是父级 Server Component 批量取数。
- 后端的"SQL 注入"→ 前端是"XSS / dangerouslySetInnerHTML"，对应 DOMPurify 消毒。
- 后端的"硬编码密钥"→ 前端要额外强调"密钥仅 server-only、禁止 NEXT_PUBLIC_ 暴露"。
- 还新增了后端没有的 **SEO 检查维度**（canonical 自指向、hreflang 双向、slug 规则），因为这是官网项目的核心质量点。

### Q4：merge-to-test 的"编译检查"前端怎么处理？没有 mvn 了。
A：前端用 `npm run lint && npm run build` 作为提测前质量门禁——TypeScript 类型检查 + 构建 + ESLint 三合一，等价于后端的编译检查。如果项目还没单测，就降级为只跑构建门禁，并提示后续补 vitest/jest。**门禁不通过不允许合并到提测分支**。
（附加细节：这个项目目录路径含中文，Turbopack 会 panic，所以 build 脚本固定带 `--webpack`。）

### Q5：hooks 是怎么工作的？pre-commit 和 post-commit 各做什么？
A：
- **pre-commit**：git commit 执行前强制触发，做代码审查 + commit message 格式校验 + 需求/Bug 编号真实性校验（调 meegle CLI 查编号是否存在），有问题阻止提交。
- **post-commit**：提交后触发，自动做 3 件事——写 learnings 经验记录、生成 test-reports 提测记录、增量更新 release-plans 发布计划。
开发者无感知，正常 commit 就全自动完成。

### Q6：经验沉淀（learnings）这套怎么形成闭环？
A：日常开发 → AI 每次修 bug/做功能自动写一条经验到 `learnings/`，随 git 同步全团队 → 下次写同类代码时 AI 先检索历史经验复用 → 周报识别重复模式 → 提炼成 steering 规则全团队生效。**写得越多越聪明**，团队知识不再只在个人脑子里。

### Q7：规范怎么版本管理和更新？
A：规范本身是一个 GitLab 仓库（`mid-platform/kiro-standards`），不是散落各项目。各项目通过"更新 kiro 规范"指令同步：clone 最新 → 对比 → 覆盖通用文件，但**保护项目个性化文件**（coding-standards、各种 `*-custom.md`、learnings、specs 等不被覆盖）。还有"自举"机制：同步时先用最新的同步规则本身，避免规则升级延迟一个版本生效。

### Q8：meegle CLI / 飞书 CLI 是干嘛的？
A：
- **meegle CLI**：飞书项目（Meego）的命令行，让 AI 能查需求/Bug 详情、校验编号、查需求配置的测试人员、在 Bug 下 @报告人追问、查排期。
- **飞书 CLI**：让 AI 读写飞书文档（读需求文档做评审、把技术方案输出到飞书）。
提测时这俩配合：从 commit 提取编号 → meegle 查测试人员姓名 → team-roster 映射飞书 open_id → 飞书机器人在群里发提测报告 + @测试人员。**信息传递从小时级降到秒级**。

### Q9：你怎么保证本地化没改坏？
A：本地化后用全局搜索复查所有 steering 文件，确认无残留的后端关键词（mvn、.java、MyBatis、@Transactional、分布式锁等）；提测门禁命令本身可跑 `npm run lint && npm run build` 验证；提交时只 stage `.kiro/` 和 `.gitignore`，确认无夹带无关文件。

### Q10：这套东西的价值/收益怎么量化？
A：规范文档给的预期收益（可作为参考目标）：需求返工 -40%、方案评审效率 +70%、低级 bug -50%、CR 人工 -50%、提测效率 +60%、发版事故 -70%、信息传递延迟从小时级到秒级。**对前端来说额外价值是把 SEO 质量点固化进了 CR 检查项**，官网项目这块最容易踩坑。

## 五、可能被追问的技术细节（防止露馅）

- **为什么 steering 用 markdown 而不是配置文件？** 因为它是给 LLM 读的提示词/规则，自然语言比结构化配置更适合表达"什么情况下该提醒什么"。
- **inclusion 字段：** steering 文件头有 `inclusion: always`（常驻）或 `inclusion: fileMatch`（匹配特定文件才加载，如 commit-convention 只在编辑代码文件时加载）。这是控制加载时机、省上下文的关键。
- **个性化文件保护机制：** 同步用"覆盖+新增"，没有删除能力，所以有"废弃文件清单"专门处理规范里删掉的文件；个性化文件在保护列表里跳过覆盖。
- **提测分支红线：** merge-to-test 严禁把开发分支合并/建 MR 到 master/main，目标分支从配置读取不写死。
- **编号防造假：** commit message 里的编号必须来自开发者本次输入或 git log 真实解析，AI 禁止从文档示例里抄编号，feat/fix 还会调 meegle 校验编号真实存在。

## 六、诚实边界（面试要把握的分寸）

- 这套规范是**后端负责人设计、并在后端项目跑通的**；你的工作是**把它改造适配到自己负责的前端项目**——这是真实且有技术含量的工作（理解规则意图、跨栈映射、补前端维度、集成落地）。
- 主推口径：**"后端负责人设计原型，我负责前端改造适配并跑通。"**（详见第七节）
- 别声称是你"从零设计了整套 AI 提效体系"，也别把后端负责人的设计说成自己的；可以说你"主导了前端侧的改造适配，并推动在项目中跑通"。
- hooks 注册、应用名、提测通知这些**部署相关项目前还没跑通**（项目未部署），如果被问要如实说"已配置好，待项目部署时启用"。

## 七、"这套东西是怎么弄出来的"——可信的由来叙事

> 真相：这是团队/组织层面共建的规范，不是你一个人从零设计的。
> 策略：**不编造个人独创**（GitLab 提交历史、团队同事一问就穿帮），而是把"这类系统真实的演化逻辑"讲透 + 把你的真实贡献讲足。讲清楚"它为什么长这样"比谎称"是我设计的"更显功力。

### 7.1 这类系统真实的演化路径（背下来，能讲就显专业）

任何团队的"AI 提效规范"基本都是这么长出来的，按这个逻辑讲，逻辑自洽、无懈可击：

1. **痛点驱动起步**：先有具体的痛——CR 靠人自觉漏得多、提测靠口头喊延迟高、同一个 bug 反复踩、发版前手忙脚乱整理清单。这些是引子。
2. **个人/小范围土办法**：有人先在自己项目里用 Kiro/Cursor 写几条 steering 规则、配个 pre-commit 检查，验证"AI 真能把这些活接管"。
3. **抽象成机制**：发现有效后，把零散规则归类——常驻规则(steering) / 按需文档(skills) / 强制触发(hooks)，这套分层是为了解决"上下文 token 成本"和"不能靠 AI 自觉"两个真问题。
4. **沉淀成仓库 + 版本化**：把规范从各项目散落收敛成一个 GitLab 仓库统一维护，做"初始化/更新规范"的同步机制，再加保护列表（个性化文件不被覆盖）、废弃清单（处理删除传播）。
5. **接入工具链闭环**：接 meegle CLI（编号校验/查测试人）、飞书机器人（提测通知@人）、统计脚本（效能度量），把"研发协作"也自动化。
6. **持续反哺**：learnings 自动沉淀 → 周报识别重复模式 → 提炼成新 steering 规则 → 全团队生效。系统自己越跑越完善。

> 一句话总结这个由来：**"它不是一次设计出来的，是从具体痛点出发、用 AI 逐步接管重复工作、再抽象分层、收敛成版本化规范仓库演化出来的。"** 这句话本身就是高质量答案。

### 7.2 你的真实角色——"后端负责人设计，我做前端改造适配"（主推口径）

> 这是最干净、最经得起追问的说法：**原型由后端负责人设计并在后端项目跑通，我把它改造适配到前端技术栈。** 归属清晰，不抢功、不夸大，同时你的"改造"工作本身有实打实的技术含量。

**问：这套规范是你设计的吗？**
> "原型是我们**后端负责人设计的**——他先在后端项目里把这套 AI 提效规范跑通（steering 规则、pre/post-commit hooks、meegle 集成那一套）。我做的是**把它改造适配到前端**：原版规则全是面向 Java/Spring 写的，直接拿过来在 Next.js 项目里基本用不了，我逐条理解它的质量意图，重写成前端等价规则，并补了前端特有的维度，再把整条链路在我项目里跑通。"

**问：那你的'改造'具体改了什么？**
> "三块：① **跨栈翻译**——分层(Controller/Service/Mapper)→组件/数据层分层、SQL注入→XSS/DOMPurify、N+1查询→组件内逐项请求、mvn编译→npm lint+build；② **补前端没有的维度**——新增 SEO 检查(canonical 自指向/hreflang 双向/slug 规则)和 TS 类型+构建门禁；③ **集成落地**——把触发文件类型、敏感文件清单、提测构建命令、meegle 配置全部换成前端项目的实际值，装授权 meegle CLI、建提测分支、跑通质量门禁。"

**问：为什么由你来做前端这块改造？**
> "我是这个前端项目的负责人，对技术栈、SEO 要求、接口契约最熟。后端负责人设计的是通用框架和后端实现，前端这块是空白，需要熟悉前端的人把它'翻译'过来并验证可行——这部分自然由我来。"

**问：你和后端负责人怎么分工的？**
> "他负责整套机制的设计和后端落地（这是源头）；我负责前端侧的适配落地。本质是**一套规范、两个技术栈的实现**，我是前端这条线的 owner。"

### 7.3 红线（用这个口径时尤其注意）

- ✅ 主推："**后端负责人设计原型，我负责前端改造适配并跑通**"——这是事实，最稳。
- ❌ 别说"是我设计的"/"是我主导整个体系"——会和"后端负责人设计"这个事实打架，自相矛盾最致命。
- ❌ 别贬低后端负责人的工作（"他那套很糙我重写的"），既不诚实也显得不会协作。
- ✅ 被深挖到机制设计层面（为什么 steering/skills/hooks 这样分）答不全时，大方说"这套机制是后端负责人设计时定的，我理解他的考量是……（讲 token 成本与强制力权衡），我主要在前端实现层面做适配"。
- ✅ 强调你"改造"的难度：**跨栈映射不是翻译文字，是理解每条规则背后的质量目标再找前端对应物**——这是你能讲出彩的地方。

### 7.4 加分项：体现你的"思考"而非"执行"

面试官真正想看的不是"你抄了一套规范"，而是"你理解它、能批判它、能扩展它"。准备 1-2 个这样的点：

- **批判性**："这套规范有个小问题——commit-convention、merge-to-test 这些文件不在同步保护列表里，下次更新规范会被覆盖掉我的前端本地化。我的处理是把易变的命令收敛到配置项，或在团队层面反馈应该支持'构建命令配置化'。"（这说明你真用过、真踩过）
- **扩展性**："前端落地后我意识到可以再加一个'前后端联调契约'的 hook——后端改了接口，自动通知前端群，这块规范里列为未来规划，我这边的接口层（lib/api 1:1 复刻契约）正好能对接。"

> 用这种"我发现了它的局限并有改进想法"的表达，比任何"是我设计的"都更有说服力。
