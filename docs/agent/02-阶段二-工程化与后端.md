## 18. Nest + LangChain 实现基于 SSE 的流式 AI 接口

### 学习目标

- 搞清楚为什么流式接口选 **SSE** 而不是 WebSocket；
- 在 Nest 里用原生 `Sse` 装饰器 / 手动写 `res` 两种方式输出 `text/event-stream`；
- 把 LangChain 的 `.stream()` 异步迭代器接到 HTTP 响应上；
- 处理好「中断、错误、结束标记」三件事。

### 核心概念讲解

**SSE（Server-Sent Events）** 是基于普通 HTTP 的单向推送协议：服务端把 `Content-Type` 设成 `text/event-stream`，然后持续往同一个连接里写 `data: xxx

` 格式的文本，浏览器用 `EventSource` 接收。它比 WebSocket 简单得多——不需要握手升级、自动重连、走标准 HTTP（对网关 / CDN / 鉴权中间件友好）。对于「模型 token 流式输出」这种**服务端单向推、客户端只收**的场景，SSE 是最佳选择。WebSocket 适合双向实时（如协同编辑、语音通话信令），用在这里属于杀鸡用牛刀。

SSE 报文格式很简单，每条消息由若干字段行组成，以一个空行结束：

```
data: 你好
data: 世界

event: done
data: [DONE]

```

LangChain 的 Chat 模型实现了 `.stream(input)`，返回一个 `AsyncIterable<AIMessageChunk>`，每个 chunk 的 `.content` 就是增量 token。我们要做的就是把这个迭代器「翻译」成 SSE 报文。

方式一：用 Nest 内置的 `@Sse()` + RxJS `Observable`（最「Nest 味」的写法）：

```ts
// src/chat/chat.controller.ts
import { Controller, Query, Sse, MessageEvent } from '@nestjs/common';
import { Observable } from 'rxjs';
import { HumanMessage, SystemMessage } from '@langchain/core/messages';
import { createChatModel } from '../llm/llm.factory';

@Controller('chat')
export class ChatController {
  // GET /chat/sse?q=讲个冷笑话
  @Sse('sse')
  sse(@Query('q') q: string): Observable<MessageEvent> {
    const model = createChatModel(true);

    return new Observable<MessageEvent>((subscriber) => {
      let aborted = false;

      (async () => {
        try {
          const stream = await model.stream([
            new SystemMessage('你是一个简洁的中文助手。'),
            new HumanMessage(q ?? '你好'),
          ]);

          for await (const chunk of stream) {
            if (aborted) break;
            const text = chunk.content?.toString() ?? '';
            if (text) subscriber.next({ data: { delta: text } });
          }
          // 自定义结束事件，前端据此关闭 EventSource
          subscriber.next({ type: 'done', data: '[DONE]' });
          subscriber.complete();
        } catch (err) {
          subscriber.error(err);
        }
      })();

      // 客户端断开时触发，停止拉取模型，省 token
      return () => {
        aborted = true;
      };
    });
  }
}
```

> 注意：`@Sse()` 返回的对象里 `data` 如果是对象，Nest 会自动 `JSON.stringify`，前端拿到的是 JSON 字符串；如果是字符串则原样输出。

方式二：拿到原生 `Response` 手动写流（更可控，适合需要精细控制 header / flush 的场景，也是和非 Nest 框架迁移时的通用写法）：

```ts
// src/chat/chat.raw.controller.ts
import { Controller, Post, Body, Res } from '@nestjs/common';
import { Response } from 'express';
import { HumanMessage } from '@langchain/core/messages';
import { createChatModel } from '../llm/llm.factory';

@Controller('chat')
export class ChatRawController {
  @Post('stream')
  async stream(@Body('q') q: string, @Res() res: Response) {
    res.set({
      'Content-Type': 'text/event-stream; charset=utf-8',
      'Cache-Control': 'no-cache, no-transform',
      Connection: 'keep-alive',
      'X-Accel-Buffering': 'no', // 关掉 Nginx 缓冲，否则不流式
    });
    res.flushHeaders();

    const model = createChatModel(true);
    const onClose = () => model /* noop, 见下文 abort */;
    res.on('close', onClose);

    try {
      const stream = await model.stream([new HumanMessage(q)]);
      for await (const chunk of stream) {
        const text = chunk.content?.toString() ?? '';
        if (text) res.write(`data: ${JSON.stringify({ delta: text })}

`);
      }
      res.write('event: done
data: [DONE]

');
    } catch (err) {
      res.write(`event: error
data: ${JSON.stringify({ message: String(err) })}

`);
    } finally {
      res.end();
    }
  }
}
```

### 小结 / 常见坑

- **Nginx 把流缓冲住了**：一定要加 `X-Accel-Buffering: no`，并在 Nginx 配 `proxy_buffering off;`，否则你会看到「转半天，最后一次性全吐出来」。
- **客户端断开不取消模型调用**：要监听 `res.on('close')` / Observable 的 teardown，置一个 `aborted` 标记跳出 `for await`，否则用户关页面了你还在烧 token。LangChain 也支持传 `signal: AbortSignal` 给 `.stream()` 做硬中断。
- **`chunk.content` 不一定是 string**：多模态模型可能返回数组，统一 `.toString()` 或做类型判断。
- **压缩中间件吃掉流**：如果全局开了 `compression()`，要对 `text/event-stream` 跳过压缩，否则不 flush。

## 19. Nest + tool 实现 OpenClaw 同款定时任务功能（上）

### 学习目标

- 理解 OpenClaw 这类「定时帮我跑 Agent」产品的后端架构；
- 用 LangChain 的 `tool()` 定义可被模型调用的工具；
- 让模型自己决定「现在该调哪个工具、传什么参数」（function calling 全流程）；
- 本节先把「工具 + Agent 执行循环」跑通，下一节再接定时调度。

### 核心概念讲解

OpenClaw 同款功能的本质是：**用户用自然语言描述一个周期性任务（"每天早上 9 点抓一下竞品价格发我邮箱"），系统把它拆成 `cron 表达式 + 一段 Agent 指令`，到点了就唤起 Agent，Agent 自主调用工具完成并回填结果。**

拆开看是两层：

1. **Agent 执行层**（本节）：给模型一组工具（HTTP 抓取、发邮件、查数据库……），模型按需调用，直到任务完成。这就是经典的 **tool calling 循环**：模型输出「我要调用 tool A，参数是 X」→ 我们执行 tool A → 把结果塞回对话 → 模型继续，直到它不再要求调工具、给出最终答案。
2. **调度层**（下节）：把「指令 + cron」持久化，用 `@nestjs/schedule` 或 BullMQ 到点触发执行层。

LangChain 用 `tool()` 把一个普通函数 + Zod schema 包装成 `StructuredTool`，模型能读懂它的名字、描述、参数结构。`createReactAgent` 或 `bindTools` + 手动循环都能驱动它。

用 LangGraph 的 `createReactAgent` 一行创建一个会用工具的 Agent（推荐，省去手写循环）：

```ts
// src/agent/agent.service.ts
import { Injectable } from '@nestjs/common';
import { createReactAgent } from '@langchain/langgraph/prebuilt';
import { HumanMessage } from '@langchain/core/messages';
import { createChatModel } from '../llm/llm.factory';
import { allTools } from './tools';

@Injectable()
export class AgentService {
  private agent = createReactAgent({
    llm: createChatModel(false), // 工具循环里通常不需要逐 token 流式
    tools: allTools,
  });

  // 运行一段自然语言指令，返回最终文本
  async run(instruction: string): Promise<string> {
    const res = await this.agent.invoke({
      messages: [new HumanMessage(instruction)],
    });
    const last = res.messages[res.messages.length - 1];
    return last.content?.toString() ?? '';
  }
}
```

如果你想**自己掌控循环**（方便埋点、限制最大步数），手写版长这样：

```ts
// src/agent/manual-loop.ts
import { HumanMessage, AIMessage, ToolMessage, BaseMessage } from '@langchain/core/messages';
import { createChatModel } from '../llm/llm.factory';
import { allTools } from './tools';

const toolMap = Object.fromEntries(allTools.map((t) => [t.name, t]));

export async function runManual(instruction: string, maxSteps = 6): Promise<string> {
  const model = createChatModel(false).bindTools(allTools);
  const messages: BaseMessage[] = [new HumanMessage(instruction)];

  for (let step = 0; step < maxSteps; step++) {
    const ai = (await model.invoke(messages)) as AIMessage;
    messages.push(ai);

    // 模型没要求调工具 → 结束
    if (!ai.tool_calls || ai.tool_calls.length === 0) {
      return ai.content?.toString() ?? '';
    }

    // 依次执行模型要求的工具调用
    for (const call of ai.tool_calls) {
      const t = toolMap[call.name];
      const output = t ? await t.invoke(call.args) : `unknown tool: ${call.name}`;
      messages.push(
        new ToolMessage({ content: String(output), tool_call_id: call.id! }),
      );
    }
  }
  return '达到最大步数仍未完成';
}
```

### 小结 / 常见坑

- **工具 description 写不好，模型就不会调**：描述要写清「什么时候用、参数啥含义」，这是 prompt engineering 的一部分。
- **Zod schema 要和描述一致**：参数名、类型、`.describe()` 都会进入模型上下文。
- **一定要设最大步数**：防止模型陷入「反复调工具」的死循环烧钱。
- **工具执行要 try/catch**：工具抛错应转成文本结果塞回去，让模型知道失败了去重试或换路，而不是整个请求 500。

## 20. Nest + tool 实现 OpenClaw 同款定时任务功能（下）

### 学习目标

- 用 `@nestjs/schedule` 实现固定 cron 任务，再用 `SchedulerRegistry` 做**动态**注册 / 取消；
- 把「指令 + cron」持久化到 Postgres，重启后自动恢复；
- 用 BullMQ + Redis 把执行从「定时器线程」挪到独立队列，避免长任务阻塞、支持重试和并发控制。

### 核心概念讲解

上节我们有了「能执行一段指令的 Agent」，本节要解决「到点自动执行」。有两种粒度：

- **轻量级（@nestjs/schedule）**：基于 cron 表达式在进程内调度。优点是零额外依赖；缺点是任务跑在主进程，长任务会占资源，且多副本部署时每个副本都会触发（需要分布式锁）。
- **生产级（BullMQ + Redis）**：调度器只负责「到点往队列里塞一个 job」，真正的 Agent 执行由 worker 消费。天然支持重试、限流、并发、多副本去重（队列保证一个 job 只被消费一次）。

OpenClaw 同款的完整链路是：**用户建任务 → 存库（指令 + cron）→ 动态注册 cron → 到点 enqueue → worker 跑 Agent → 结果回写 + 通知用户**。

动态 cron 注册（`@nestjs/schedule` + `SchedulerRegistry`）：

```ts
// src/jobs/scheduler.service.ts
import { Injectable, OnModuleInit } from '@nestjs/common';
import { SchedulerRegistry } from '@nestjs/schedule';
import { CronJob } from 'cron';
import { InjectRepository } from '@nestjs/typeorm';
import { Repository } from 'typeorm';
import { InjectQueue } from '@nestjs/bullmq';
import { Queue } from 'bullmq';
import { AgentTask } from './task.entity';

@Injectable()
export class SchedulerService implements OnModuleInit {
  constructor(
    private readonly registry: SchedulerRegistry,
    @InjectRepository(AgentTask) private readonly repo: Repository<AgentTask>,
    @InjectQueue('agent') private readonly queue: Queue,
  ) {}

  // 服务启动时，从库里恢复所有 enabled 任务
  async onModuleInit() {
    const tasks = await this.repo.find({ where: { enabled: true } });
    tasks.forEach((t) => this.register(t));
  }

  register(task: AgentTask) {
    if (this.registry.doesExist('cron', task.id)) this.unregister(task.id);

    const job = new CronJob(task.cron, async () => {
      // 到点不直接执行，而是入队，交给 worker（解耦 + 可重试）
      await this.queue.add('run-task', { taskId: task.id }, {
        attempts: 3,
        backoff: { type: 'exponential', delay: 5000 },
        removeOnComplete: 100,
      });
    });

    this.registry.addCronJob(task.id, job as any);
    job.start();
  }

  unregister(id: string) {
    if (this.registry.doesExist('cron', id)) {
      this.registry.getCronJob(id).stop();
      this.registry.deleteCronJob(id);
    }
  }

  // 新建任务：存库 + 立即注册
  async create(dto: Pick<AgentTask, 'name' | 'instruction' | 'cron'>) {
    const task = await this.repo.save(this.repo.create({ ...dto, enabled: true }));
    this.register(task);
    return task;
  }
}
```

BullMQ worker（真正跑 Agent 的地方）：

```ts
// src/jobs/agent.processor.ts
import { Processor, WorkerHost } from '@nestjs/bullmq';
import { Job } from 'bullmq';
import { InjectRepository } from '@nestjs/typeorm';
import { Repository } from 'typeorm';
import { AgentTask } from './task.entity';
import { AgentService } from '../agent/agent.service';

@Processor('agent', { concurrency: 2 }) // 最多 2 个任务并发跑
export class AgentProcessor extends WorkerHost {
  constructor(
    @InjectRepository(AgentTask) private readonly repo: Repository<AgentTask>,
    private readonly agent: AgentService,
  ) {
    super();
  }

  async process(job: Job<{ taskId: string }>) {
    const task = await this.repo.findOneByOrFail({ id: job.data.taskId });
    const result = await this.agent.run(task.instruction); // 复用第 19 节的 Agent
    await this.repo.update(task.id, { lastResult: result });
    return result; // 失败时抛错，BullMQ 会按 attempts/backoff 自动重试
  }
}
```

2. 配好 TypeORM 连接（`TypeOrmModule.forRoot`），`synchronize: true` 先让它自动建表（生产用 migration）。
3. 写个 `POST /tasks` 调 `scheduler.create()`，建一个 `cron: '* * * * *'`（每分钟）、`instruction: '查北京天气并通知我'` 的任务。
4. 等一分钟，看 worker 日志和 `lastResult` 字段；重启服务，确认 `onModuleInit` 把任务恢复了。

### 小结 / 常见坑

- **多副本重复触发**：`@nestjs/schedule` 在每个副本都会跑 cron。解决办法就是「cron 只负责入队，BullMQ 保证 job 唯一消费」——这正是上面架构的价值；或给 cron 加 Redis 分布式锁。
- **长任务别放在 cron 回调里直接 await**：会阻塞调度线程。一律入队。
- **cron 表达式时区**：`CronJob` 默认用机器时区，跨时区部署要显式传 `timeZone: 'Asia/Shanghai'`。
- **Agent 执行可能很慢 / 失败**：靠 BullMQ 的 `attempts` + `backoff` 重试，并设 `removeOnComplete` 防止 Redis 堆积。

## 21. 给 Agent 加上语音交互：ASR + 流式 TTS

### 学习目标

- 理解一条语音对话链路：**录音 → ASR（语音转文字）→ Agent → TTS（文字转语音）→ 播放**；
- 用 OpenAI 兼容的 `audio.transcriptions`（Whisper）做 ASR；
- 用流式 TTS 边生成边播，降低首音延迟；
- 在 Nest 里处理音频文件上传和音频流响应。

### 核心概念讲解

语音 Agent 的延迟体验取决于三段的串联方式：

- **ASR（Automatic Speech Recognition）**：把用户音频转成文字。可整段转（等用户说完上传整个音频），也可流式转（边说边出字，体验更好但接口更复杂）。入门先做整段。
- **LLM 推理**：拿到文字喂给第 19 节的 Agent。
- **TTS（Text-To-Speech）**：把回答转成音频。**流式 TTS** 是关键——不要等整段文字生成完再合成，而是 LLM 吐出一句就合成一句、推一句，用户听到首音的时间能从「几秒」降到「几百毫秒」。

OpenAI 兼容接口里：ASR 用 `POST /audio/transcriptions`（multipart 上传音频），TTS 用 `POST /audio/speech`（返回音频字节流，支持 `response_format: 'mp3' | 'pcm'` 等）。

### 关键代码

```ts
import { Injectable } from '@nestjs/common';
import OpenAI from 'openai';
import { Response } from 'express';

@Injectable()
export class TtsService {
  private client = new OpenAI({
    apiKey: process.env.OPENAI_API_KEY,
    baseURL: process.env.OPENAI_BASE_URL,
  });

  // 把一段文字合成 mp3，流式写入 HTTP response
  async speakToResponse(text: string, res: Response) {
    const resp = await this.client.audio.speech.create({
      model: process.env.TTS_MODEL ?? 'tts-1',
      voice: 'alloy',
      input: text,
      response_format: 'mp3',
    });

    res.set({ 'Content-Type': 'audio/mpeg', 'Cache-Control': 'no-cache' });
    // resp.body 是一个 Web ReadableStream，转 Node 流后 pipe
    const nodeStream = (await import('stream')).Readable.fromWeb(resp.body as any);
    nodeStream.pipe(res);
  }
}
```

```ts
// src/voice/voice.controller.ts
import {
  Controller, Post, UploadedFile, UseInterceptors, Res,
} from '@nestjs/common';
import { FileInterceptor } from '@nestjs/platform-express';
import { Response } from 'express';
import { AsrService } from './asr.service';
import { TtsService } from './tts.service';
import { AgentService } from '../agent/agent.service';

@Controller('voice')
export class VoiceController {
  constructor(
    private readonly asr: AsrService,
    private readonly tts: TtsService,
    private readonly agent: AgentService,
  ) {}

  // 前端把录音以 multipart 字段 'audio' 上传
  @Post('chat')
  @UseInterceptors(FileInterceptor('audio'))
  async chat(@UploadedFile() file: Express.Multer.File, @Res() res: Response) {
    const userText = await this.asr.transcribe(file.buffer, file.originalname);
    const reply = await this.agent.run(userText);
    // 把回复文本放到 header，方便前端同时显示字幕
    res.set('X-Reply-Text', encodeURIComponent(reply));
    await this.tts.speakToResponse(reply, res); // 直接返回语音流
  }
}
```

### 小结 / 常见坑

- **音频格式**：浏览器 `MediaRecorder` 常出 `audio/webm;codecs=opus`，Whisper 能直接吃；若用别的 ASR 服务可能要先转码（ffmpeg）。

- **延迟优化优先级**：先优化 TTS 流式（占体感大头），再考虑流式 ASR。

## 22. AGUI 协议：Vercel AI SDK + LangChain 实现流式组件渲染

### 学习目标

- 理解 **AGUI（Agent-GUI）** 的核心理念：后端不只流式文本，还流式「该渲染什么 UI」；
- 用 **Vercel AI SDK** 的 `streamText` / `useChat` 打通前后端流式；
- 让 LangChain 后端通过 Data Stream Protocol 把工具调用、自定义数据推给前端做组件渲染。

### 核心概念讲解

传统流式接口只推纯文本。但 Agent 时代我们想要更丰富的交互：模型调用「查天气」工具后，前端直接渲染一个天气卡片而不是一段文字；模型在「思考」时前端显示 loading；模型返回结构化数据时前端渲染表格。

**AGUI 的理念**就是把「Agent 的过程和产物」结构化地流给前端，由前端决定渲染成什么组件。**Vercel AI SDK** 是目前落地这套理念最顺手的工具：

- 后端用 `streamText()` 返回一个标准的流式响应（Data Stream Protocol，本质是带类型前缀的 SSE：`0:` 文本、`9:` 工具调用、`a:` 工具结果、`2:` 自定义 data……）；
- 前端用 `useChat()` Hook 自动解析这个流，`messages` 里带 `parts`（文本片段、工具调用片段），`data` 里带自定义数据，直接 map 成组件。

LangChain 和 Vercel AI SDK 可以共存：要么用 AI SDK 的 `@ai-sdk/openai` provider（最顺），要么把 LangChain 的流**适配**成 AI SDK 的 Data Stream 格式（用 `@ai-sdk/langchain` 适配器 `toDataStreamResponse`）。

### 关键代码

后端（Nest 里挂一个走 AI SDK 协议的路由）。先看纯 AI SDK 写法：

```bash

```

```ts
import { z } from 'zod';

  apiKey: process.env.OPENAI_API_KEY,
  baseURL: process.env.OPENAI_BASE_URL,
});

export class AguiController {
  @Post('chat')

    const result = streamText({
      model: openai(process.env.CHAT_MODEL ?? 'gpt-4o-mini'),
      messages,
      tools: {
        getWeather: tool({
          description: '查询城市天气，返回结构化数据供前端渲染卡片',

            // 返回结构化对象，前端拿到后渲染天气卡片
```

```ts
import { toDataStreamResponse } from '@ai-sdk/langchain';

      ['system', '你是中文助手'],
      ['human', '{q}'],

    nodeStream.pipe(res);
  }
}
```

- **版本差异**：AI SDK v4/v5 的 API（`parts` 结构、`pipeDataStreamToResponse` 名称）有变化，以你安装版本的官方文档为准；本文按 v4 心智模型书写。

## 23. 图编排引擎：LangGraph 和多 Agent 架构

### 学习目标

- 理解 LangGraph 的核心抽象：**State（状态）/ Node（节点）/ Edge（边）/ 条件边**；
- 用 `StateGraph` 手写一个有「分支 + 循环」的图，看清 Agent 循环的本质；
- 搭一个 **Supervisor（主管）多 Agent** 架构：一个路由 Agent 把任务分派给多个专家 Agent。

### 核心概念讲解

`createReactAgent`（第 19 节）够用，但当流程变复杂——「先检索，检索不到就改写问题再检索，最多三次；拿到资料再生成；生成完做事实校验，不通过就重写」——这种带分支和循环的逻辑用链式 `.pipe()` 很难表达。**LangGraph** 把 Agent 流程建模成**有向图**：

- **State**：一个贯穿全图的共享状态对象（通常含 `messages` 数组）。每个节点读它、返回部分更新，框架用 **reducer** 合并（比如 messages 用 append 而不是覆盖）。
- **Node**：一个函数 `(state) => Partial<State>`，代表一步操作（调模型、调工具、做判断）。
- **Edge**：节点间的连接。普通边是「A 跑完一定去 B」；**条件边**是「A 跑完根据返回值决定去 B 还是 C 还是 END」——这就是分支和循环的来源。

多 Agent 架构常见两种：

- **Supervisor**：一个主管节点负责「看当前状态，决定下一步交给哪个专家」，专家干完回到主管，主管再决策，直到完成。
- **Swarm / 网状**：Agent 之间直接互相移交（handoff）。

入门掌握 Supervisor 即可。

### 关键代码

装依赖：


先手写一个最小图，体会 State/Node/条件边：

```ts
// src/graph/basic-graph.ts
import { StateGraph, Annotation, START, END } from '@langchain/langgraph';
import { BaseMessage, HumanMessage } from '@langchain/core/messages';
import { ToolNode } from '@langchain/langgraph/prebuilt';
import { createChatModel } from '../llm/llm.factory';
import { allTools } from '../agent/tools';

// 1. 定义 State：messages 用「追加」reducer
const GraphState = Annotation.Root({
  messages: Annotation<BaseMessage[]>({
    reducer: (prev, next) => prev.concat(next),
    default: () => [],
  }),
});

const model = createChatModel(false).bindTools(allTools);

// 2. 节点：调模型
async function callModel(state: typeof GraphState.State) {
  const res = await model.invoke(state.messages);
  return { messages: [res] };
}

// 3. 条件边：模型要调工具 → 去 tools 节点；否则 → END
function shouldContinue(state: typeof GraphState.State) {
  const last: any = state.messages[state.messages.length - 1];
  return last.tool_calls?.length ? 'tools' : END;
}

// 4. 组图
const graph = new StateGraph(GraphState)
  .addNode('agent', callModel)
  .addNode('tools', new ToolNode(allTools))
  .addEdge(START, 'agent')
  .addConditionalEdges('agent', shouldContinue, { tools: 'tools', [END]: END })
  .addEdge('tools', 'agent') // 工具跑完回到 agent —— 这就是「循环」
  .compile();

export async function runGraph(q: string) {
  const out = await graph.invoke({ messages: [new HumanMessage(q)] });
  return out.messages[out.messages.length - 1].content?.toString();
}
```

> 上面这张「agent ↔ tools 来回跳，直到不再调工具」的图，本质就是 `createReactAgent` 内部干的事。看懂它你就理解了所有 Agent 循环。

再看 **Supervisor 多 Agent**：一个主管，两个专家（数学专家 / 写作专家）：

```ts
// src/graph/supervisor.ts
import { StateGraph, Annotation, START, END } from '@langchain/langgraph';
import { BaseMessage, HumanMessage, AIMessage } from '@langchain/core/messages';
import { z } from 'zod';
import { createChatModel } from '../llm/llm.factory';

const State = Annotation.Root({
  messages: Annotation<BaseMessage[]>({
    reducer: (a, b) => a.concat(b),
    default: () => [],
  }),
  next: Annotation<string>({ reducer: (_, b) => b, default: () => '' }),
});

const members = ['math_expert', 'writer'] as const;

// 主管：用结构化输出决定下一个去谁，或 FINISH
async function supervisor(state: typeof State.State) {
  const router = createChatModel(false).withStructuredOutput(
    z.object({ next: z.enum([...members, 'FINISH']) }),
  );
  const decision = await router.invoke([
    {
      role: 'system',
      content:
        `你是主管。可调度的专家：${members.join(', ')}。` +
        `根据对话决定下一步交给谁；任务完成则回 FINISH。`,
    },
    ...state.messages,
  ]);
  return { next: decision.next };
}

// 专家节点工厂
function makeExpert(name: string, persona: string) {
  return async (state: typeof State.State) => {
    const llm = createChatModel(false);
    const res = await llm.invoke([
      { role: 'system', content: persona },
      ...state.messages,
    ]);
    return { messages: [new AIMessage({ content: res.content as string, name })] };
  };
}

const graph = new StateGraph(State)
  .addNode('supervisor', supervisor)
  .addNode('math_expert', makeExpert('math_expert', '你只解决数学计算问题，给出步骤和答案。'))
  .addNode('writer', makeExpert('writer', '你只负责把内容润色成通顺中文。'))
  .addEdge(START, 'supervisor')
  // 专家干完都回到主管，让主管重新决策
  .addEdge('math_expert', 'supervisor')
  .addEdge('writer', 'supervisor')
  // 主管的条件边：按 next 字段路由
  .addConditionalEdges('supervisor', (s) => (s.next === 'FINISH' ? END : s.next), {
    math_expert: 'math_expert',
    writer: 'writer',
    [END]: END,
  })
  .compile();

export async function runSupervisor(q: string) {
  const out = await graph.invoke({ messages: [new HumanMessage(q)] });
  return out.messages.map((m) => `${(m as any).name ?? m.getType()}: ${m.content}`);
}
```

### 动手步骤

1. 跑 `runGraph('查北京天气并通知我')`，在 `callModel` 里打日志，观察 agent → tools → agent 的循环次数。
2. 跑 `runSupervisor('帮我算 (23*17)+5，然后把结果写成一句通顺的话')`，看主管先派给 `math_expert` 再派给 `writer` 最后 FINISH。
3. 用 `graph.stream(...)` 替代 `invoke`，逐节点观察中间状态（LangGraph 的可观测性优势）。
4. 给图加 `checkpointer`（如 `MemorySaver`）实现多轮记忆 / 断点续跑。

### 小结 / 常见坑

- **reducer 一定要写对**：`messages` 必须用 append 语义，否则每个节点会覆盖掉历史，Agent 立刻失忆。
- **条件边的返回值要和 mapping 的 key 一一对应**，拼错字符串会静默走不到任何分支。
- **死循环**：Supervisor 若一直不返回 FINISH 会无限跑，用 `graph.invoke(input, { recursionLimit: 25 })` 设上限兜底。
- **专家共享同一份 messages**：要不要让专家看到全部历史 vs 只看主管转发的子任务，是架构设计点，影响成本和效果。

## 24. Agentic RAG：基于 LangGraph 实现大模型自主决策的 RAG 闭环系统

### 学习目标

- 区分**传统 RAG** 和 **Agentic RAG**：后者让模型自主决定「要不要检索、检索什么、检索结果够不够、要不要改写重检」；
- 用 LangGraph 把「检索 → 评分 → （改写重检 | 生成）」连成一个带循环的闭环图；
- 这是「企业级知识库 Agent」的核心引擎雏形。

### 核心概念讲解

**传统 RAG** 是固定流水线：`用户问题 → 向量检索 top-k → 拼进 prompt → 生成`。问题是：检索质量差时模型只能硬编，问题表述刁钻时检索召回为零。

**Agentic RAG** 引入「自主决策」：

1. **要不要检索**：寒暄类问题直接答，不浪费检索；
2. **检索后做相关性评分（grade）**：拿到的文档真的相关吗？
3. **不相关就改写问题（rewrite）再检索**：换个说法、补充关键词，最多重试 N 次；
4. **足够相关才生成（generate）**，并可选地做事实校验。

用 LangGraph 表达这套「带评分和循环」的逻辑最自然。下面实现一个经典闭环：`retrieve → grade →（相关? generate : rewrite → retrieve）`。

### 关键代码

向量库这里用内存版演示（生产换 pgvector / Milvus，见阶段三）：

```bash
pnpm add @langchain/langgraph @langchain/openai @langchain/core langchain
```

```ts
// src/rag/agentic-rag.ts
import { StateGraph, Annotation, START, END } from '@langchain/langgraph';
import { OpenAIEmbeddings, ChatOpenAI } from '@langchain/openai';
import { MemoryVectorStore } from 'langchain/vectorstores/memory';
import { Document } from '@langchain/core/documents';
import { z } from 'zod';

const llm = new ChatOpenAI({
  apiKey: process.env.OPENAI_API_KEY,
  model: process.env.CHAT_MODEL ?? 'gpt-4o-mini',
  configuration: { baseURL: process.env.OPENAI_BASE_URL },
});
const embeddings = new OpenAIEmbeddings({
  apiKey: process.env.OPENAI_API_KEY,
  model: process.env.EMBED_MODEL ?? 'text-embedding-3-small',
  configuration: { baseURL: process.env.OPENAI_BASE_URL },
});

// 准备一个简单知识库
let store: MemoryVectorStore;
export async function initStore() {
  store = await MemoryVectorStore.fromDocuments(
    [
      new Document({ pageContent: '公司年假政策：入职满 1 年享 5 天带薪年假，每多 1 年加 1 天，上限 15 天。' }),
      new Document({ pageContent: '报销流程：登录 OA 提交发票，主管审批后 3 个工作日到账。' }),
    ],
    embeddings,
  );
}

// 1. State
const RagState = Annotation.Root({
  question: Annotation<string>({ reducer: (_, b) => b }),
  docs: Annotation<string[]>({ reducer: (_, b) => b, default: () => [] }),
  tries: Annotation<number>({ reducer: (_, b) => b, default: () => 0 }),
  answer: Annotation<string>({ reducer: (_, b) => b, default: () => '' }),
});

// 2. 节点：检索
async function retrieve(state: typeof RagState.State) {
  const results = await store.similaritySearch(state.question, 3);
  return { docs: results.map((d) => d.pageContent), tries: state.tries + 1 };
}

// 3. 节点：相关性评分（结构化输出 yes/no）
async function grade(state: typeof RagState.State): Promise<Partial<typeof RagState.State>> {
  return {}; // 评分结果通过条件边读取，这里不改 state；真实可把分数写进 state
}
async function gradeDecision(state: typeof RagState.State) {
  const grader = llm.withStructuredOutput(
    z.object({ relevant: z.boolean().describe('检索到的文档能否回答问题') }),
  );
  const { relevant } = await grader.invoke(
    `问题：${state.question}
文档：
${state.docs.join('
')}
这些文档能回答问题吗？`,
  );
  if (relevant) return 'generate';
  if (state.tries >= 3) return 'generate'; // 重试上限，兜底直接生成
  return 'rewrite';
}

// 4. 节点：改写问题
async function rewrite(state: typeof RagState.State) {
  const res = await llm.invoke(
    `原问题检索不到资料，请把它改写得更适合检索（补充同义词/关键词），只输出改写后的问题：
${state.question}`,
  );
  return { question: res.content?.toString() ?? state.question };
}

// 5. 节点：生成
async function generate(state: typeof RagState.State) {
  const res = await llm.invoke(
    `仅根据以下资料回答问题，资料不足就说不知道。
资料：
${state.docs.join('
')}

问题：${state.question}`,
  );
  return { answer: res.content?.toString() ?? '' };
}

// 6. 组图：retrieve → (grade 条件) → generate / rewrite→retrieve
const graph = new StateGraph(RagState)
  .addNode('retrieve', retrieve)
  .addNode('grade', grade)
  .addNode('rewrite', rewrite)
  .addNode('generate', generate)
  .addEdge(START, 'retrieve')
  .addEdge('retrieve', 'grade')
  .addConditionalEdges('grade', gradeDecision, {
    generate: 'generate',
    rewrite: 'rewrite',
  })
  .addEdge('rewrite', 'retrieve') // 改写后回去重检 —— 闭环
  .addEdge('generate', END)
  .compile();

export async function ask(question: string) {
  const out = await graph.invoke({ question });
  return out.answer;
}
```

### 动手步骤

1. 跑 `demo.ts`，在 `gradeDecision` 和 `rewrite` 里打日志，观察「钱怎么报？」是否触发了一次改写（改成「报销流程怎么走」之类）再命中。
2. 故意问知识库里没有的问题（「公司养几只猫？」），看是否在重试 3 次后兜底回答「不知道」而不是死循环。
3. 把 `MemoryVectorStore` 换成 pgvector（阶段三会详细讲），其余图结构不变——这就是图编排的好处：检索后端可替换。
4. 给图加一个 `generate` 后的「事实校验」节点，不通过则回 `rewrite`，扩展成更严格的闭环。

### 小结 / 常见坑

- **没有重试上限会死循环**：`tries` 计数 + 条件边里的兜底分支是必须的。
- **评分太严会一直重写**：grade 的 prompt 要平衡，宁可放进生成节点让模型自己判断「资料不足就说不知道」。
- **改写问题要约束输出**：明确「只输出改写后的问题」，否则模型会输出一堆解释污染下一轮检索。
- **Agentic RAG 比传统 RAG 贵**：多了评分、改写的额外调用。对延迟/成本敏感的场景可以只在「首轮检索召回为空」时才启用 Agentic 路径。

## 25. 基于 Docker Compose 的本地开发提效和生产环境部署

### 学习目标

- 用一份 `docker-compose.yml` 一键拉起 Agent 项目的全部依赖：Postgres(pgvector) / Redis / ElasticSearch / Milvus；
- 理解「本地开发 compose」和「生产部署 compose」的差异（数据卷、健康检查、资源限制、`.env` 分离）；
- 把 Nest 应用打包成镜像，多阶段构建瘦身。

### 核心概念讲解

一个企业级知识库 Agent 至少依赖 4 个中间件：关系库（业务数据 + pgvector 向量）、Redis（队列 + 缓存）、ElasticSearch（全文检索）、Milvus（大规模向量）。手动 `docker run` 四个容器再处理网络互联很痛苦，**Docker Compose** 把这些声明成一个文件，`docker compose up -d` 一把起，容器间用服务名互相访问（如 `postgres:5432`）。

本地开发 vs 生产的关键差异：

- **本地**：端口全映射到宿主机方便调试、`synchronize`/热重载开着、数据卷可随时删；
- **生产**：只暴露必要端口、加 `healthcheck` + `depends_on: condition: service_healthy` 保证启动顺序、加 `restart: unless-stopped`、敏感配置走 `.env` 文件而非写死、加资源 `limits`。

### 关键代码

依赖中间件的 compose（`docker-compose.infra.yml`，本地开发用）：

5. 上生产时 `docker compose -f docker-compose.infra.yml -f docker-compose.prod.yml up -d --build`，app 容器用服务名连中间件。

### 小结 / 常见坑

- **app 用 `localhost` 连不上中间件**：容器内 `localhost` 是它自己。容器间必须用**服务名**（`postgres`、`redis`）。
- **ES 起不来 / OOM**：Linux 要 `sysctl -w vm.max_map_count=262144`，并给 `ES_JAVA_OPTS` 限内存。
- **数据卷别用匿名卷**：命名卷（如 `pgdata`）才能 `docker compose down` 后数据还在；`down -v` 会删卷，慎用。
- **生产别关 ES 安全**：`xpack.security.enabled=false` 只限本地；生产要开认证 + TLS。
- **启动顺序**：`depends_on` 不带 `condition` 只保证「启动先后」不保证「就绪」，一定配 `healthcheck` + `service_healthy`。

## 26. ElasticSearch 全文检索：倒排索引表 + IK 分词器 + BM25 算法

### 学习目标

- 理解全文检索三大基石：**倒排索引、分词、相关性打分（BM25）**；
- 给中文检索装 **IK 分词器**，对比 `ik_max_word` 和 `ik_smart`；
- 在 Node 里用 `@elastic/elasticsearch` 建索引、写文档、查询，看懂 `_score`；
- 为阶段三的「BM25 + 向量」混合检索打基础。

### 核心概念讲解

**倒排索引（Inverted Index）** 是搜索引擎的核心数据结构。正排是「文档 → 它包含哪些词」，倒排反过来：「词 → 哪些文档包含它，以及位置/词频」。

```
词项「年假」 → [doc1: 2次, doc5: 1次]
词项「报销」 → [doc2: 1次, doc7: 3次]
```

查询「年假」时直接查表拿到 doc 列表，O(1) 级定位，不用扫全部文档。这就是 ES 快的根本原因。

**分词（Analyzer）** 决定「文档和查询怎么切成词项」。英文按空格天然分词，**中文不行**——"我要查年假政策" 必须切成「我 / 要 / 查 / 年假 / 政策」。ES 默认的 `standard` 分词器对中文是按单字切，效果差。**IK 分词器**是中文社区标准：

- `ik_max_word`：最细粒度，「中华人民共和国」切成「中华人民共和国/中华人民/中华/华人/人民共和国/人民/共和国/共和/国」，召回高，适合建索引；
- `ik_smart`：粗粒度，切成「中华人民共和国」，适合查询。

**BM25** 是 ES 默认的相关性打分算法（Lucene 实现）。它在 TF-IDF 基础上改进，核心三因素：

- **词频 TF**：词在文档里出现越多越相关，但有**饱和**（出现 100 次不比 10 次相关 10 倍）；
- **逆文档频率 IDF**：越罕见的词区分度越高（「的」到处都有，权重低；「pgvector」很稀有，权重高）；
- **文档长度归一化**：长文档天然含更多词，要惩罚，避免长文档总是赢。

`_score` 就是 BM25 算出来的相关性分数，越高越靠前。

```ts
import { es } from './es.client';

const INDEX = 'kb_docs';

export async function createIndex() {
  const exists = await es.indices.exists({ index: INDEX });
  if (exists) return;

  await es.indices.create({
    index: INDEX,
    settings: {
      // 可选：调 BM25 参数 k1/b
      similarity: { default: { type: 'BM25', k1: 1.2, b: 0.75 } },
    },
    mappings: {
      properties: {
        title: { type: 'text', analyzer: 'ik_max_word', search_analyzer: 'ik_smart' },
        content: { type: 'text', analyzer: 'ik_max_word', search_analyzer: 'ik_smart' },
        category: { type: 'keyword' }, // keyword 不分词，用于精确过滤/聚合
      },
    },
  });
}
```

```ts
// src/search/query.ts
import { es } from './es.client';

export async function search(q: string) {
  const res = await es.search({
    index: 'kb_docs',
    query: {
      multi_match: {
        query: q,
        fields: ['title^2', 'content'], // title 权重 ×2
        type: 'best_fields',
      },
    },
    highlight: { fields: { content: {} } }, // 高亮命中片段
  });

  return res.hits.hits.map((h) => ({
    score: h._score,          // BM25 分数
    title: (h._source as any).title,
    highlight: h.highlight?.content?.[0],
  }));
}
```

### 小结 / 常见坑

- **IK 版本必须和 ES 完全一致**：8.13.4 的 ES 只能配 8.13.4 的 IK，版本错插件加载失败。
- **建索引和查询用不同分词器**：`analyzer: ik_max_word`（索引时细切高召回）+ `search_analyzer: ik_smart`（查询时粗切高精度）是常见最佳实践。
- **`text` vs `keyword`**：要全文检索用 `text`（会分词）；要精确匹配/过滤/聚合用 `keyword`（不分词）。一个字段可以同时建两种（`fields` 子字段）。
- **mapping 建好不能改分词器**：要改得 reindex。设计时想清楚。
- **BM25 不懂语义**：它只匹配字面词项，「番茄」搜不到「西红柿」。这正是阶段三要做「BM25 + 向量混合检索」的原因——字面匹配和语义匹配互补。
