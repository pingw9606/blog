## 35. Nest 进阶：企业级 Node.js 后端最主流框架

### 学习目标

- 理解 Nest 的依赖注入（DI）容器到底解决了什么问题，掌握 provider 作用域、自定义 provider、`forwardRef` 循环依赖。
- 能写出并组合 **拦截器（Interceptor）/ 守卫（Guard）/ 管道（Pipe）/ 异常过滤器（Filter）** 这四类横切关注点。
- 用 `@nestjs/config` 做多环境配置（`.env.development` / `.env.production`），并对配置做校验。
- 实现一套 JWT 鉴权 + 角色守卫，保护 Agent 接口。
- 接入结构化日志（pino），把 trace 串起来。

### 核心概念讲解

#### 35.1 为什么需要依赖注入

前面写 demo 时，我们到处 `new LangChainService()`、`new MilvusClient()`。这在脚本里没问题，但服务一大就出现：

- **耦合**：A 模块直接 `new` 了 B，换实现要改 A 的源码。
- **难测试**：想 mock 掉真实的向量库调用？你得想办法替换掉 `new`。
- **生命周期混乱**：一个数据库连接池本该全局单例，结果被 `new` 出来一堆。

Nest 的 DI 容器把「创建对象」和「使用对象」分开。你只声明「我需要一个 `EmbeddingService`」，容器负责实例化、缓存、注入。换实现、写测试、管理单例都变成配置问题，而不是改代码。

DI 的三个核心角色：

- **Provider**：能被注入的东西（service、repository、工厂、值）。用 `@Injectable()` 标记。
- **Module**：组织边界。`providers` 里注册、`exports` 里导出给别的模块用、`imports` 里引入别人导出的。
- **Token**：注入的「钥匙」。默认用类本身当 token，也可以用字符串/Symbol 自定义 token（注入接口、配置值时常用）。

#### 35.2 Provider 作用域（Scope）

| 作用域 | 行为 | 用途 |
|--------|------|------|
| `Scope.DEFAULT`（单例） | 全应用一个实例 | 绝大多数 service、连接池 |
| `Scope.REQUEST` | 每个请求一个实例 | 需要拿到请求上下文（如 per-request 的 user / traceId） |
| `Scope.TRANSIENT` | 每次注入都新建 | 有状态、不想共享的轻量对象 |

注意：REQUEST 作用域会「传染」——依赖它的 provider 也会变成 request 作用域，有性能成本，别滥用。

#### 35.3 横切关注点：守卫 / 管道 / 拦截器 / 过滤器

一个请求进 Nest，执行顺序大致是：

```
请求
  → 中间件 Middleware
  → 守卫 Guard          (能不能进来？鉴权/权限 → 返回 false 抛 403)
  → 拦截器 Interceptor  (前置：改请求 / 计时开始)
  → 管道 Pipe           (校验 + 转换入参，如 DTO 校验)
  → 控制器 Controller / Service  (业务逻辑)
  → 拦截器 Interceptor  (后置：包装响应 / 计时结束 / 日志)
  → 异常过滤器 Filter   (任何环节抛错都在这里统一兜底)
响应
```

记住这个顺序，你就知道每类横切逻辑该放哪：
- **鉴权、限流** → Guard
- **入参校验、类型转换** → Pipe
- **统一响应格式、计时、缓存、日志** → Interceptor
- **统一错误响应** → Filter

#### (1) 自定义 Provider + 注入接口（DI 进阶）

接口在运行时会被 TS 擦除，不能当注入 token，所以用字符串/Symbol token：

```ts
// src/llm/llm.tokens.ts
export const LLM_CLIENT = Symbol('LLM_CLIENT');

export interface LLMClient {
  chat(prompt: string): Promise<string>;
}
```

```ts
// src/llm/openai-llm.service.ts
import { Injectable } from '@nestjs/common';
import OpenAI from 'openai';
import type { LLMClient } from './llm.tokens';

@Injectable()
export class OpenAILLMService implements LLMClient {
  private readonly client = new OpenAI({
    apiKey: process.env.OPENAI_API_KEY,
    baseURL: process.env.OPENAI_BASE_URL, // OpenAI 兼容网关，如 https://api.openai.com/v1
  });

  async chat(prompt: string): Promise<string> {
    const res = await this.client.chat.completions.create({
      model: process.env.LLM_MODEL ?? 'gpt-4o-mini',
      messages: [{ role: 'user', content: prompt }],
    });
    return res.choices[0]?.message?.content ?? '';
  }
}
```

```ts
// src/llm/llm.module.ts
import { Module } from '@nestjs/common';
import { LLM_CLIENT } from './llm.tokens';
import { OpenAILLMService } from './openai-llm.service';

@Module({
  providers: [
    // 用 Symbol token 绑定接口到具体实现，换实现只改这一行
    { provide: LLM_CLIENT, useClass: OpenAILLMService },
  ],
  exports: [LLM_CLIENT],
})
export class LlmModule {}
```

```ts
// 用 zod 定义配置 schema，启动时校验，缺 key 直接崩，避免运行到一半才发现没配
export const envSchema = z.object({
  NODE_ENV: z.enum(['development', 'production', 'test']).default('development'),
  PORT: z.coerce.number().default(3000),
  OPENAI_API_KEY: z.string().min(1),

export function validateEnv(config: Record<string, unknown>): Env {
  const parsed = envSchema.safeParse(config);
  if (!parsed.success) {
    throw new Error(
      'Invalid environment variables:
' +
        JSON.stringify(parsed.error.format(), null, 2),
    );
  }
  return parsed.data;
}
```

```ts
// src/auth/jwt.strategy.ts
import { Injectable } from '@nestjs/common';
import { PassportStrategy } from '@nestjs/passport';
import { ExtractJwt, Strategy } from 'passport-jwt';

export interface JwtPayload {
  sub: string;
  username: string;
  roles: string[];
```

```ts
// src/auth/roles.guard.ts
import { CanActivate, ExecutionContext, Injectable } from '@nestjs/common';
import { Reflector } from '@nestjs/core';
import { ROLES_KEY } from './roles.decorator';

@Injectable()
export class RolesGuard implements CanActivate {
  constructor(private reflector: Reflector) {}

  canActivate(context: ExecutionContext): boolean {
    const required = this.reflector.getAllAndOverride<string[]>(ROLES_KEY, [
      context.getHandler(),
      context.getClass(),
    ]);
    if (!required || required.length === 0) return true; // 没标 @Roles 就放行
    const { user } = context.switchToHttp().getRequest();
    return required.some((r) => user?.roles?.includes(r));
  }
}
```

```ts
// src/chat/chat.controller.ts
import { Body, Controller, Post, UseGuards } from '@nestjs/common';
import { JwtAuthGuard } from '../auth/jwt-auth.guard';

@Controller('chat')
```

#### (5) 拦截器：统一响应 + 计时

```ts
// src/common/transform.interceptor.ts
import {
  CallHandler,
  ExecutionContext,
  Injectable,
  NestInterceptor,
  Logger,
} from '@nestjs/common';
import { Observable } from 'rxjs';
import { map, tap } from 'rxjs/operators';

@Injectable()
export class TransformInterceptor<T> implements NestInterceptor<T, unknown> {
  private readonly logger = new Logger('HTTP');

  intercept(ctx: ExecutionContext, next: CallHandler<T>): Observable<unknown> {
    const start = Date.now();
    const req = ctx.switchToHttp().getRequest();
    return next.handle().pipe(
      tap(() =>
        this.logger.log(`${req.method} ${req.url} - ${Date.now() - start}ms`),
      ),
      // 统一包一层 { code, data, timestamp }
      map((data) => ({ code: 0, data, timestamp: Date.now() })),
    );
  }
}
```

#### (6) 异常过滤器：统一错误格式

```ts
// src/common/all-exceptions.filter.ts
import {
  ArgumentsHost,
  Catch,
  ExceptionFilter,
  HttpException,
  HttpStatus,
  Logger,
} from '@nestjs/common';

@Catch()
export class AllExceptionsFilter implements ExceptionFilter {
  private readonly logger = new Logger('Exception');

  catch(exception: unknown, host: ArgumentsHost) {
    const res = host.switchToHttp().getResponse();
    const status =
      exception instanceof HttpException
        ? exception.getStatus()
        : HttpStatus.INTERNAL_SERVER_ERROR;
    const message =
      exception instanceof HttpException
        ? exception.getResponse()
        : 'Internal server error';

    this.logger.error(exception instanceof Error ? exception.stack : exception);
    res.status(status).json({
      code: status,
      message,
      timestamp: Date.now(),
    });
  }
}
```

### 小结 / 常见坑

- **忘记 `reflect-metadata`**：`main.ts` 顶部（或 polyfill）必须 `import 'reflect-metadata'`，否则装饰器元数据读不到，DI 全挂。
- **接口不能当注入 token**：TS 接口运行时不存在，必须用 class、字符串或 Symbol token + `@Inject()`。
- **Guard/Pipe/Interceptor 执行顺序记混**：守卫最先（决定能不能进），管道在控制器前校验入参，拦截器包裹前后，过滤器兜底。
- **REQUEST 作用域传染**：一个 request 作用域 provider 会让依赖链上游全变成 request 作用域，注意性能。
- **配置不校验**：生产环境最痛的 bug 往往是「某个 env 没配」。务必用 zod/Joi 启动即校验，fail fast。
- **循环依赖**：A 注入 B、B 又注入 A 时用 `forwardRef(() => B)`，但更应该反思设计是否该拆出第三方。

## 36. 图解 Transformer 架构：大模型底层原理

### 学习目标

- 讲清 **自注意力（Self-Attention）** 中 Q、K、V 三个矩阵分别是什么、怎么算出注意力权重。
- 理解 **多头注意力（Multi-Head Attention）** 为什么要拆成多个头。
- 理解 **位置编码（Positional Encoding）** 解决了什么问题。
- 用一段可运行的 TypeScript 实现单头注意力，把公式落到代码。
- 直观理解「为什么参数规模大了会涌现（emergence）能力」。

### 核心概念讲解

#### 36.1 一句话总览

Transformer 干的事：**把一串 token 的向量序列，反复用「注意力 + 前馈网络」加工，让每个 token 的向量都融合进它该关注的上下文信息**。GPT 这类只用解码器（decoder-only），核心模块堆叠 N 层，每层结构相同。

单层结构（decoder block）：

```
输入向量序列 X  (shape: [seq_len, d_model])
   │
   ├─► [多头自注意力] ──► 残差相加 + LayerNorm
   │
   └─► [前馈网络 FFN]  ──► 残差相加 + LayerNorm
   │
输出向量序列  (shape 不变, 喂给下一层)
```

#### 36.2 注意力机制：QKV 到底是什么

类比图书馆检索：

- **Query（查询）**：我手里这个 token「想找什么信息」。
- **Key（键）**：每个 token「自我介绍我有什么信息」。
- **Value（值）**：每个 token「真正能提供的内容」。

计算流程：拿我的 Q 去和**所有** token 的 K 做点积 → 得到「我对每个 token 的相关度分数」→ softmax 归一化成权重 → 用权重对所有 V 加权求和 → 得到这个 token「融合了上下文后的新向量」。

公式（Scaled Dot-Product Attention）：

```
Attention(Q, K, V) = softmax( Q · Kᵀ / √d_k ) · V
```

其中 `√d_k` 是缩放因子：维度越大点积值越大，softmax 会过于尖锐导致梯度消失，除以 `√d_k` 把方差拉回来。

Q、K、V 怎么来？输入向量 X 分别乘三个**可学习**的权重矩阵 `W_Q`、`W_K`、`W_V`：

```
Q = X · W_Q
K = X · W_K
V = X · W_V
```

模型训练，本质就是在学这些 W 矩阵——学会「什么样的词该关注什么样的词」。

> **因果掩码（causal mask）**：GPT 生成下一个词时只能看左边，不能偷看右边的答案。所以在 softmax 前，把当前位置右侧的分数设成 `-∞`，softmax 后权重变 0。

#### 36.3 多头注意力：为什么要拆头

一组 QKV 只能学到一种「关注模式」。多头就是把 `d_model` 切成 `h` 份，每份独立算一套注意力，最后拼回来。好处：

- **head 1** 可能学到「关注语法主谓关系」，**head 2** 关注「指代消解」，**head 3** 关注「邻近词」……
- 类似 CNN 用多个卷积核提取不同特征。

```
MultiHead(Q,K,V) = Concat(head_1, ..., head_h) · W_O
其中 head_i = Attention(Q·W_Q^i, K·W_K^i, V·W_V^i)
```

`d_model = 512, h = 8` 时，每个头维度 `d_k = 64`。

#### 36.4 位置编码：注意力本身「没有顺序感」

注意力是对集合做加权求和，打乱 token 顺序结果不变——但语言显然有顺序（「狗咬人」≠「人咬狗」）。所以要把位置信息注入向量。

经典做法（原论文，正弦位置编码）：

```
PE(pos, 2i)   = sin(pos / 10000^(2i/d_model))
PE(pos, 2i+1) = cos(pos / 10000^(2i/d_model))
```

不同维度用不同频率的正余弦，让模型能通过线性运算推出相对位置。现代模型（LLaMA、GPT-NeoX）多用 **RoPE（旋转位置编码）**，把位置信息以旋转的方式直接作用在 Q、K 上，外推性更好。

#### 36.5 前馈网络与残差

注意力之后接一个两层 MLP（FFN），对每个位置独立做非线性变换：`FFN(x) = W₂ · GELU(W₁·x + b₁) + b₂`，中间维度通常是 `4 × d_model`。注意力负责「token 之间交换信息」，FFN 负责「单个 token 内部加工 / 记忆知识」。

残差连接（`x + Sublayer(x)`）和 LayerNorm 保证几十上百层堆叠时梯度能稳定回传。

#### 36.6 为什么会「涌现」

涌现（emergent abilities）指：某些能力（如多步推理、in-context learning）在小模型上几乎为零，模型规模（参数量 / 数据量 / 算力）越过某个阈值后突然大幅提升，不是平滑增长。

可以这样直观理解（学界仍在研究，以下是主流解释）：

- **容量阈值**：复杂能力（比如「先分类再按类执行」）需要足够多的「电路」组合。参数不够时根本装不下这套电路，表现接近随机；够了之后能力一下子可用。
- **压缩即智能**：预测下一个 token 这个简单目标，逼着模型为了降低 loss 去学习语法、事实、推理、甚至世界模型。规模越大，能压缩进去的规律越多。
- **度量错觉的争议**：也有研究指出部分「涌现」是评测指标非线性（如严格 exact-match）造成的错觉，换成连续指标会变平滑。但 in-context learning 等能力的阶跃仍被广泛观察到。

对你的实际意义：**别用小模型硬刚需要推理的任务**（换更大模型或拆解任务）；**用 few-shot 示例激发 in-context learning**；理解了注意力，你就知道为什么「把关键信息放 prompt 开头/结尾」、「上下文太长会注意力稀释」是真实存在的现象。

### 关键代码

下面用纯 TypeScript 实现**单头自注意力 + 因果掩码**，不依赖任何深度学习框架，帮你把公式跑起来看清每一步。

```ts
// attention.ts —— 最小可运行的单头自注意力
// 用纯数组实现矩阵运算，重在看清流程，不追求性能

type Matrix = number[][];

// 矩阵乘法 A(m×n) · B(n×p) = (m×p)
function matmul(a: Matrix, b: Matrix): Matrix {
  const m = a.length, n = b.length, p = b[0].length;
  const out: Matrix = Array.from({ length: m }, () => new Array(p).fill(0));
  for (let i = 0; i < m; i++)
    for (let k = 0; k < n; k++)
      for (let j = 0; j < p; j++) out[i][j] += a[i][k] * b[k][j];
  return out;
}

// 转置
function transpose(a: Matrix): Matrix {
  return a[0].map((_, j) => a.map((row) => row[j]));
}

// 按行 softmax
function softmaxRows(a: Matrix): Matrix {
  return a.map((row) => {
    const max = Math.max(...row);
    const exps = row.map((v) => Math.exp(v - max)); // 减最大值防溢出
    const sum = exps.reduce((s, v) => s + v, 0);
    return exps.map((v) => v / sum);
  });
}

// 单头自注意力：输入 X (seq_len × d_model)
function selfAttention(
  X: Matrix,
  Wq: Matrix,
  Wk: Matrix,
  Wv: Matrix,
  causal = true,
): Matrix {
  const Q = matmul(X, Wq); // (seq × d_k)
  const K = matmul(X, Wk);
  const V = matmul(X, Wv);
  const dk = Q[0].length;

  // scores = Q · Kᵀ / √d_k
  let scores = matmul(Q, transpose(K));
  scores = scores.map((row) => row.map((s) => s / Math.sqrt(dk)));

  // 因果掩码：位置 i 不能看 j > i
  if (causal) {
    for (let i = 0; i < scores.length; i++)
      for (let j = 0; j < scores[i].length; j++)
        if (j > i) scores[i][j] = -Infinity;
  }

  const weights = softmaxRows(scores); // 每行是「该 token 对各位置的注意力权重」
  return matmul(weights, V);           // 加权求和得到新向量
}

// ---- demo ----
// 3 个 token，每个 4 维
const X: Matrix = [
  [1, 0, 1, 0],
  [0, 2, 0, 2],
  [1, 1, 1, 1],
];
// 随机权重（真实模型中由训练得到），这里 d_model=4, d_k=4
const rand = (r: number, c: number): Matrix =>
  Array.from({ length: r }, () =>
    Array.from({ length: c }, () => Math.random() - 0.5),
  );

const out = selfAttention(X, rand(4, 4), rand(4, 4), rand(4, 4));
console.log('attention output:');
console.table(out);
```

运行：

```bash
npx tsx attention.ts
```

你会看到输出仍是 `3 × 4` 矩阵——每个 token 的向量被它「该关注的上下文」重新加工过了。把 `causal` 设为 `false` 再跑，对比权重分布的变化。

### 动手步骤

1. 跑通上面的脚本，打印 `weights` 矩阵，观察因果掩码后下三角才有值。
2. 把 `X` 的第 2、3 行对调，验证「没有位置编码时注意力对顺序不敏感」（输出只是对应行也跟着换，相对关系不变）。
3. 进阶：用 Python + HuggingFace `transformers` 加载一个小模型（如 `gpt2`），用 `output_attentions=True` 取出真实注意力矩阵，可视化某层某头到底在关注哪些词——把理论和真实模型对上。

### 小结 / 常见坑

- **QKV 不是三种不同的数据，而是同一份输入经过三个不同投影矩阵**得到的三种「视角」。
- **缩放 `√d_k` 不能省**，否则高维时 softmax 退化、训练不稳。
- **位置编码是「加」进 embedding 的（或在 RoPE 中作用于 QK）**，不是拼接。
- 别把「涌现」当玄学营销词——它有具体含义（能力随规模的非线性跃升），也有「指标错觉」的争议，面试时能讲清两面会加分。
- 理解注意力是 O(seq²) 复杂度，就明白为什么长上下文又贵又慢，以及 FlashAttention、稀疏注意力在优化什么。

### 三、数据流

**入库流（Ingestion）**

```
上传文件 → Nest Ingestion 接收并存元数据(PG)
  → LangChain Loader 解析(PDF/DOCX/MD)
  → Splitter 切分成 chunks
  → 对每个 chunk:
       ├─ embedding → 写入 Milvus/pgvector  (语义检索)
       ├─ 原文 → 写入 ElasticSearch         (关键词/全文检索)
       └─ LLM 抽取实体关系 → 写入 Neo4j      (知识图谱)
  → 更新 PG 中文档状态为 "ready"
```

**问答流（Query）**

```
用户提问 → Nest Chat(鉴权/校验)
  → LangGraph 入口
     1. Router: 判断是闲聊 / 知识问答 / 需要多跳推理
     2. Retrieve: 并行查 Milvus(向量)+ES(全文) → RRF 融合 → Reranker 重排 Top-K
        (若涉及实体关系，补查 Neo4j 子图)
     3. 读 Mem0 取该用户长期记忆，拼进上下文
     4. Generate: LLM 基于检索结果 + 记忆生成答案(带引用)
     5. Reflect: 自检答案是否被证据支撑；不足则回到第 2 步重检索(最多 N 轮)
  → 流式返回前端(SSE)，同时:
       ├─ 写对话历史(PG)
       ├─ 更新 Mem0(若产生新结论/偏好)
       └─ LangSmith 记录整条 trace
```

### 五、分阶段实施计划

按「先跑通主链路，再加高级特性」推进，每阶段都有可演示成果：

**阶段 A：地基（第 1 周）**
- 用 Nest 搭好骨架：配置校验、JWT 鉴权、统一日志/响应/异常。
- `docker-compose` 起齐 PG / Redis / Milvus。
- 验收：登录拿到 token，访问受保护接口返回统一格式。

**阶段 B：最小 RAG 闭环（第 2 周）**
- Ingestion：支持 PDF/MD 上传 → 切分 → embedding → Milvus。
- Retrieval：纯向量检索 Top-K。
- Chat：检索 + LLM 生成，前端用 Vercel AI SDK 流式展示。
- 验收：上传一份文档，问其中内容能拿到带引用的正确答案。

**阶段 C：混合检索 + 重排（第 3 周）**
- 接入 ElasticSearch 全文检索，与向量做 RRF 融合。
- 加 Reranker 重排，提升 Top-K 精度。
- 接入 Neo4j：抽取实体关系，支持「X 和 Y 是什么关系」类问题。
- 验收：对比纯向量 vs 混合检索的命中率（用 LangSmith 评测集）。

**阶段 D：Agent 化 + 记忆（第 4 周）**
- 用 LangGraph 把流程改成有状态图：Router → Retrieve → Generate → Reflect 自纠循环。
- 接入 Mem0：记住用户偏好与历史结论，跨会话生效。
- 验收：一个需要「检索不足→重检索」的复杂问题能自动多轮收敛。

**阶段 E：可观测 + 上线（第 5 周）**
- 全链路接 LangSmith：trace、token 成本、延迟、评测。
- 加限流（Redis）、缓存、错误告警。
- 容器化部署，配 `.env.production`。
- 验收：能在 LangSmith 看到每次问答的完整链路与成本，压测下服务稳定。

**进阶（可选）**
- 用 DeepAgents 拆解超复杂任务（如「总结这 50 份合同的风险点并生成报告」）为子 Agent。
- 多租户隔离、文档级权限（不同角色看不同知识库）。

### 小结 / 常见坑

- **别一上来追求大而全**：严格按阶段 A→E 推进，每阶段都保持「可演示」，比憋一个月最后跑不起来强得多。
- **检索质量决定上限**：80% 的回答问题其实是检索没召回对的文档，优先打磨 Ingestion 切分策略和混合检索，而不是反复换更大的 LLM。
- **可观测要趁早接**：阶段 B 就把 LangSmith 接上，否则后期排查「为什么这个问题答错了」会非常痛苦。
- **记忆别滥用**：Mem0 只存真正需要跨会话的偏好/结论，把整段对话塞进记忆会污染上下文、推高成本。
- **鉴权和多租户要在设计期就考虑**：知识库往往涉及敏感数据，文档级权限后补极其麻烦。
