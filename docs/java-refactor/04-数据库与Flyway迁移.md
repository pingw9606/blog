# 04 · 数据库与 Flyway 迁移

讲清三件事：本地怎么零配置跑（H2）、生产怎么连 PostgreSQL、表结构怎么用 Flyway 规范管理（对应旧项目的 prisma migrate）。

## 一、多 profile：一套代码，多种环境

用 Spring 的 profile 区分环境。`application.yml` 定公共部分 + 默认 profile：

```yaml
spring:
  profiles:
    active: local        # 默认本地
  flyway:
    enabled: false       # 默认关，H2 用 JPA 建表；postgres/prod 再开
app:
  jwt:
    secret: ${APP_JWT_SECRET:dev-only-secret-at-least-32-bytes-long-xxxxxx}
    expiration-ms: 604800000
  cors:
    allowed-origins: ${APP_CORS_ORIGINS:http://localhost:5173}
```

### local：H2 内存库，零配置

`application-local.yml`：

```yaml
spring:
  datasource:
    url: jdbc:h2:mem:ouyang;DB_CLOSE_DELAY=-1;MODE=PostgreSQL
    driver-class-name: org.h2.Driver
    username: sa
    password: ""
  jpa:
    hibernate:
      ddl-auto: create-drop   # 启动建表、停机删表
  h2:
    console:
      enabled: true           # http://localhost:8080/h2-console
```

好处：`mvn spring-boot:run` 直接能跑，不用先装数据库。学习期效率最高。

### postgres / prod：连真库 + Flyway

`application-postgres.yml`：

```yaml
spring:
  datasource:
    url: ${DATABASE_URL:jdbc:postgresql://localhost:5432/ouyang}
    username: ${DB_USER:ouyang}
    password: ${DB_PASSWORD:ouyang_dev_pwd}
  jpa:
    hibernate:
      ddl-auto: validate      # 只校验，不自动改表；建表交给 Flyway
  flyway:
    enabled: true
```

> `ddl-auto` 取值：`create-drop`（每次重建，仅测试）、`update`（自动补表，方便但危险）、`validate`（只校验，生产推荐）、`none`。**生产别用 update**——它可能在你没预期时改表。

## 二、Flyway：把建表变成可追溯的迁移

旧项目用 `prisma migrate`；Java 侧的等价物是 **Flyway**。加依赖：

```xml
<dependency><groupId>org.flywaydb</groupId><artifactId>flyway-core</artifactId></dependency>
<dependency><groupId>org.flywaydb</groupId><artifactId>flyway-database-postgresql</artifactId></dependency>
```

迁移脚本放 `src/main/resources/db/migration/`，命名 `V1__init.sql`、`V2__xxx.sql`……启动时按版本号顺序执行，执行过的记在 `flyway_schema_history` 表里，不会重复跑。

### 怎么写出和实体完全一致的 V1

难点：`ddl-auto=validate` 要求表结构和实体**精确匹配**，手写容易差类型。**技巧：先让 Hibernate 自己建一遍，再把 DDL 导出来当迁移脚本。**

```bash
# 1. 建个临时干净库
psql -d postgres -c "CREATE DATABASE ouyang_ddl OWNER ouyang;"
# 2. 用 update 让 JPA 建表（临时关 flyway）
DATABASE_URL=jdbc:postgresql://localhost:5432/ouyang_ddl \
  java -jar target/ouyang-server-*.jar \
  --spring.profiles.active=postgres \
  --spring.flyway.enabled=false --spring.jpa.hibernate.ddl-auto=update
# 3. 导出 schema（Ctrl-C 停掉应用后）
pg_dump -U ouyang -d ouyang_ddl --schema-only --no-owner --no-privileges > schema.sql
```

把 `schema.sql` 整理成 `V1__init.sql`（约束名可换成可读的）。片段：

```sql
CREATE TABLE users (
    id            bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    email         varchar(255) NOT NULL,
    username      varchar(255) NOT NULL,
    password_hash varchar(255) NOT NULL,
    role          varchar(255) NOT NULL,
    status        varchar(255) NOT NULL,
    created_at    timestamp(6) with time zone,
    updated_at    timestamp(6) with time zone,
    CONSTRAINT uk_users_email UNIQUE (email),
    CONSTRAINT users_role_check CHECK (role IN ('reader','author','admin'))
);
-- novels / volumes / chapters / comments / author_applications / reading_progress ...
```

> Hibernate 的 `validate` 只校验表和列（类型/是否可空），**不校验约束名和索引名**，所以可以把 `uk6dotkott...` 这种随机名改成 `uk_users_email` 这种可读名。

## 三、验证 Flyway 生效

对一个全新库启动 postgres profile，日志应看到：

```
Migrating schema "public" to version "1 - init"
Successfully applied 1 migration to schema "public", now at version v1
```

再查 `flyway_schema_history`：

```bash
psql -U ouyang -d ouyang -c "select version, description, success from flyway_schema_history;"
#  version | description | success
#  1       | init        | t
```

应用能正常启动（说明 `validate` 通过），注册接口能写入数据，就对了。

## 四、以后改表怎么办

**不要**去改 `V1__init.sql`（已执行的迁移不能改，Flyway 会校验 checksum）。新增一个 `V2__add_xxx.sql`：

```sql
-- V2__add_novel_word_count.sql
ALTER TABLE novels ADD COLUMN word_count integer NOT NULL DEFAULT 0;
```

同时在实体里加对应字段。下次启动 Flyway 自动执行 V2。
