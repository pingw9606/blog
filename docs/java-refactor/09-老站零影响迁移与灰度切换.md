# 09 · 老站零影响迁移与灰度切换

前 8 篇是"从零重构 + 部署流水线"。这篇讲最难的一步：把一个**有真实用户和数据、正在对外运行的 Nuxt 老站**，迁到新的 Java + React 栈，且**对用户零影响**——原密码能登、内容不丢、几乎不断服。

> 背景：老站是 Nuxt 全栈（Prisma + PostgreSQL），跑在一台服务器上，有 5 个用户、6 部小说、63 章真实数据。

## 一、先想清楚"零影响"意味着什么

| 维度 | 要求 |
|------|------|
| 账号 | 老用户用**原密码**能登，不用重置 |
| 内容 | 小说/章节/评论/进度**完整保留** |
| 服务 | 切换窗口尽量短，可**随时回滚** |

三个技术难点，逐个拆。

## 二、难点一：密码跨栈兼容（scrypt → bcrypt）

老站 `nuxt-auth-utils` 用的是 **scrypt**（底层 `@adonisjs/hash`），哈希是 PHC 格式：

```
$scrypt$n=16384,r=8,p=1$<base64(salt)>$<base64(hash)>
```

新栈用 **bcrypt**（`$2a$...`）。两者格式不通用——直接迁过去，老用户密码全部失效。

**解法：写一个兼容两种格式的 `PasswordEncoder`**。登录时按前缀判断用哪种校验；验证通过后，如果是旧 scrypt 哈希，就顺手重算成 bcrypt 存回去（**对用户完全无感的渐进式升级**）。

Java 侧用 Bouncy Castle 的 `SCrypt` 复现（参数要和 adonisjs 默认一致：N=16384, r=8, p=1, keyLength=64）：

```java
public class ScryptCompatPasswordEncoder implements PasswordEncoder {
    private final BCryptPasswordEncoder bcrypt = new BCryptPasswordEncoder();
    private static final Pattern PHC =
        Pattern.compile("^\\$scrypt\\$n=(\\d+),r=(\\d+),p=(\\d+)\\$([^$]+)\\$([^$]+)$");

    @Override public String encode(CharSequence raw) { return bcrypt.encode(raw); } // 新密码统一 bcrypt

    @Override public boolean matches(CharSequence raw, String encoded) {
        if (encoded.startsWith("$scrypt$")) return matchesScrypt(raw, encoded);
        return bcrypt.matches(raw, encoded);
    }
    public boolean isLegacyScrypt(String e) { return e != null && e.startsWith("$scrypt$"); }

    private boolean matchesScrypt(CharSequence raw, String phc) {
        Matcher m = PHC.matcher(phc);
        if (!m.matches()) return false;
        int n = Integer.parseInt(m.group(1)), r = Integer.parseInt(m.group(2)), p = Integer.parseInt(m.group(3));
        byte[] salt = b64(m.group(4)), expected = b64(m.group(5));
        byte[] actual = SCrypt.generate(raw.toString().getBytes(UTF_8), salt, n, r, p, expected.length);
        return MessageDigest.isEqual(actual, expected); // 常量时间比较
    }
    // PHC 用无 padding 的 base64，解码时补齐
    private static byte[] b64(String s) { return Base64.getDecoder().decode(s + "=".repeat((4 - s.length() % 4) % 4)); }
}
```

登录成功后升级：

```java
if (passwordEncoder instanceof ScryptCompatPasswordEncoder sc && sc.isLegacyScrypt(user.getPasswordHash())) {
    user.setPasswordHash(passwordEncoder.encode(rawPassword)); // 转 bcrypt
    userRepository.save(user);
}
```

> **关键：一定要写跨语言一致性测试**。做法是：先用老项目的 node 代码对已知明文（如 `password123`）生成一条真实哈希，再在 Java 单测里断言这条哈希能被校验通过。两边算法/参数只要差一点，测试立刻挂——比上线后才发现登不上强太多。

```java
// 这条哈希由老项目 node 代码对 "password123" 生成
static final String LEGACY = "$scrypt$n=16384,r=8,p=1$0qGVEPcgTAM9o5ySbqwM1g$ABCu3Vs8...";
@Test void verifiesLegacyScryptHash() {
    assertTrue(encoder.matches("password123", LEGACY));
    assertFalse(encoder.matches("wrong", LEGACY));
}
```

## 三、难点二：数据迁移（Prisma 表 → JPA 表，同库共存）

一个很省事的发现：**两套表能在同一个 Postgres 库里并存**——Prisma 默认表名是 PascalCase 带引号（`"User"`、`"Novel"`），JPA 我们用 snake_case（`users`、`novels`），**不冲突**。

于是迁移就是同库内一段 `INSERT ... SELECT`，不用 dump 来 dump 去：

```sql
BEGIN;
-- 列名映射：Prisma 驼峰 → JPA 下划线；枚举 ::text；时间按 UTC 还原为 timestamptz
INSERT INTO users (id, email, username, password_hash, role, status, created_at, updated_at)
SELECT id, email, username, "passwordHash", role::text, status::text,
       "createdAt" AT TIME ZONE 'UTC', "updatedAt" AT TIME ZONE 'UTC'
FROM "User";
-- novels / volumes / chapters / comments / author_applications / reading_progress 同理 ...

-- 显式带过来了主键，最后把自增序列推到 max(id)，避免新插入冲突
SELECT setval(pg_get_serial_sequence('users','id'), COALESCE((SELECT max(id) FROM users), 1));
-- ...其余表同理
COMMIT;
```

几个易错点：
- **枚举**：Prisma 的枚举是 PG `USER-DEFINED` 类型，插进 `varchar` 前要 `::text`。两边字符串值一致（reader/draft/published…）所以兼容。
- **时间**：Prisma 存的是 `timestamp without time zone`（UTC），插进 `timestamptz` 要 `AT TIME ZONE 'UTC'` 才是正确时刻。
- **只 INSERT 到新表**，不碰 Prisma 老表 → 对老站**无损、可并存**。

## 四、难点三：Flyway 遇到非空库

新栈用 Flyway 建表（见 [04](./04-数据库与Flyway迁移.md)）。但这个库里**已经有 Prisma 的表**，Flyway 默认拒绝在非空 schema 上迁移：

```
Found non-empty schema(s) "public" but no schema history table.
```

解法：允许基线，且**基线版本设为 0**——这样我们的 `V1` 仍会执行（建出小写表），又不因已有表而报错：

```yaml
spring:
  flyway:
    enabled: true
    baseline-on-migrate: true
    baseline-version: 0     # 关键：设 0，否则会跳过 V1
```

启动日志应看到 `baselined ... version: 0` 后紧接着 `Migrating ... to version "1"`。

## 五、灰度切换与回滚

有了上面三块，切换就很稳。原则：**先并存验证，最后一步才切流量**。

1. **备份**：`pg_dump` 存一份，切换前的保命符。
2. **并存起新栈**：让 Java 后端连**同一个** Postgres（同库），Flyway 建小写表；跑迁移 SQL 灌数据；新前端起在一个**备用端口**（如 8081），老站继续占着 80。
3. **验证**：在服务器 `curl 127.0.0.1:8081` 走一遍——小说列表、章节、登录。
4. **切流量**：停老站的 nginx（腾出 80）→ 新前端容器换到 `0.0.0.0:80`。窗口只有几秒。
5. **回滚**：老容器只是**停掉、没删**，出问题一条命令切回：
   ```bash
   docker rm -f 新前端容器 && docker start 老nginx 老app
   ```

> 老站的数据表也原样留着。真正确认新站稳定、跑了一阵子之后，再清理老容器和老表不迟。**"能回滚"比"切得快"重要。**

## 六、一条真实的架构坑

部署时撞到一个典型问题：**开发机是 Apple Silicon（arm64），服务器是 x86_64（amd64）**。本地 `docker build` 出来的镜像架构不对，拿到服务器跑不了。

两条路：
- 用 **CI（GitHub Actions，ubuntu-latest 是 amd64）产出的镜像**，服务器拉下来跑；
- 或直接**在服务器上 build**（把源码传上去 `docker build`）。

另外，`docker compose`（带空格的插件）和 `docker-compose`（独立二进制）在不同机器上装的可能不一样，脚本里注意区分。

## 小结

零影响迁移 = **密码兼容层 + 同库共存迁移 + Flyway 基线 + 灰度切换/回滚**。每一块单独都不难，难在"组合起来还要保证线上用户无感"。核心心法就一句：**先并存、可回滚，最后才切流量**。
